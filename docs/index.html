How It Works

Right Shift
→
Speak
→
Right Shift
→
Text at cursor

VoiceNative sits in your menu bar. When you press Right Shift, it captures audio from your microphone, transcribes it locally using WhisperKit on Apple's Neural Engine, and pastes the result directly into whatever app you're working in.

Everything runs on-device. Audio never leaves your Mac. No API keys, no accounts, no recurring costs.

What to Expect

First launch downloads the Whisper model (~950 MB) from Hugging Face and compiles it for the Neural Engine. This takes 60–90 seconds. It happens once. Every launch after loads from cache in about 5 seconds.

Recording limit is 5 minutes. This is a deliberate default — the same limit macOS dictation uses. For recordings over 30 seconds, a background pipeline transcribes in chunks while you speak, so you don't wait at the end.

Model keepalive: the Neural Engine can unload CoreML models after periods of inactivity. VoiceNative runs a micro-inference (silent audio) every 2 minutes to keep the pipeline warm. Without this, the first transcription after idle can take 30+ seconds instead of milliseconds.

A 10-second recording transcribes in about 3 seconds. A 2-minute recording, with streaming, finishes within seconds of you pressing stop.

Audio capture runs on AVAudioEngine's real-time thread. The buffer is lock-free. Resampling and normalization happen on the transcription task's thread, never on the audio callback. This separation prevents audio glitches during transcription.

Choosing the Right Model

Building a local voice transcription tool sounds simple. Record audio, run Whisper, paste text. The reality was a sequence of problems where each solution revealed the next constraint. It started with the model.

Two real options exist on macOS:

whisper.cpp

C/C++ implementation. GGML model format. Runs on Metal (GPU). Loads instantly — just a file read. The ecosystem standard.

WhisperKit

Apple's CoreML integration. Purpose-built for Neural Engine (ANE) on Apple Silicon. Swift-native. Ships as an SPM package.

We chose WhisperKit. The Neural Engine is a dedicated inference accelerator that Apple includes in every M-series chip. It runs machine learning models without competing for GPU or CPU time. For a menu bar app that needs to transcribe in the background without affecting the user's workflow, that matters.

The trade-off: CoreML requires a compilation step. The first time you load a Whisper model through CoreML, it compiles the neural network graph specifically for your hardware's Neural Engine configuration. This takes 60–90 seconds. whisper.cpp + Metal loads in under a second.

We initially measured 97 seconds for the first model load. After identifying the correct cache path (~/Documents/huggingface/models/argmaxinc/whisperkit-coreml), subsequent loads dropped to ~5 seconds. The difference between "broken" and "fast" was a single directory path.

The model is openai_whisper-large-v3_turbo — a 955 MB quantized variant. It balances transcription quality against memory footprint. We tried smaller models; they hallucinated more on short utterances. We tried unquantized; memory climbed without meaningful accuracy improvement.

The Neural Engine Problem

Choosing the Neural Engine opened a set of problems that don't exist with GPU inference.

Compute unit selection

WhisperKit lets you choose compute units for each model component: the audio encoder and the text decoder. The options are CPU-only, CPU+GPU, or CPU+Neural Engine.

Our first working configuration used cpuAndGPU for the encoder. It loaded fast, transcribed correctly, and seemed fine. Then we switched to cpuAndNeuralEngine.

The compute configuration that stuck melCompute: .cpuAndGPU audioEncoderCompute: .cpuAndNeuralEngine textDecoderCompute: .cpuAndNeuralEngine prefillCompute: .cpuOnly The mel spectrogram runs on GPU (fast, no ANE compilation). Encoder and decoder run on the Neural Engine (optimized for sustained inference). Prefill stays on CPU (small overhead, avoids ANE contention).

The transcription speed improvement was significant. But we discovered a critical constraint:

Switching compute units — even once — invalidates the CoreML cache. If you load with GPU encoder, then switch to ANE encoder, CoreML recompiles from scratch. Another 90-second wait. The cache is keyed by the exact compute unit configuration.

We tried splitting: GPU for the encoder (fast load), ANE for the decoder (fast inference). The moment you mix compute units across model components, CoreML sometimes recompiles the full graph. We abandoned the hybrid approach and committed to full ANE.

The keepalive problem

After the model loads, it stays in Neural Engine memory — until macOS decides to reclaim it. We observed the ANE silently unloading the model after a few minutes of inactivity. The next transcription would then take 30+ seconds as CoreML reloaded and re-optimized.

The fix: a keepalive timer that runs a micro-inference (transcribing an empty 0.5-second buffer) every 2 minutes. This keeps the CoreML/ANE pipeline warm without visible side effects.

/// Run a micro-inference to keep CoreML/ANE pipeline warm.
func prewarm() async {
    let silence = [Float](repeating: 0, count: 8000)
    let options = DecodingOptions(
        task: .transcribe, language: "en",
        withoutTimestamps: true, suppressBlank: true
    )
    _ = try? await whisperKit.transcribe(
        audioArray: silence, decodeOptions: options
    )
}

We also register for NSWorkspace.didWakeNotification to prewarm after the Mac wakes from sleep — another scenario where ANE unloads models.

The Streaming Pipeline

Whisper's architecture has a hard limit: it processes a maximum of 30 seconds of audio per inference pass. If you record for 2 minutes and wait until the end to transcribe, the user sits through four sequential 30-second passes. At a real-time factor of 0.3, that's ~36 seconds of waiting.

The insight: you can transcribe chunks while the user is still speaking.

Record
→
Every 30s: transcribe chunk in background
→
Stop
→
Only transcribe remaining tail

The pipeline runs on a 5-second polling loop. When accumulated raw audio exceeds 30 seconds, it snapshots that chunk, resamples it to 16kHz, and sends it to WhisperKit. The raw sample index advances past the chunk to prevent re-transcription.

The math

Without streaming (worst case) Wait time = ceil(duration / 30) × 30 × RTF A 2-minute recording: ceil(120/30) × 30 × 0.3 = 36 seconds of post-stop waiting.
With streaming Wait time = (duration mod 30) × RTF Same 2-minute recording: (120 mod 30) × 0.3 = 0 seconds. The pipeline caught up completely. In practice, 0–9 seconds depending on where in the 30s cycle you stop.

On stop, the completed chunk texts are concatenated with whatever remains in the tail. The result: a 2-minute recording feels nearly instant.

The streaming pipeline was the single largest perceived-performance improvement. The actual transcription speed didn't change — we just moved the work from "after you stop" to "while you're still talking."

Audio Engineering

Whisper expects 16kHz mono Float32 audio. macOS microphones typically run at 48kHz. The path between them matters more than it seems.

Capture at native rate, resample on demand

We use AVAudioEngine configured at the microphone's native sample rate (typically 48kHz). The raw samples accumulate in a thread-safe buffer. When a chunk or the final recording needs transcription, we resample to 16kHz mono using AVAudioConverter.

An alternative approach captures directly at 16kHz using CoreAudio's AUHAL unit. This avoids the resampling step entirely. We chose AVAudioEngine because it handles device switching gracefully and doesn't require low-level CoreAudio buffer management.

Normalization

After resampling, the audio is peak-normalized using Apple's Accelerate framework (vDSP). This ensures consistent input levels regardless of microphone gain settings:

// Find peak amplitude
var peak: Float = 0
vDSP_maxmgv(samples, 1, &peak, vDSP_Length(samples.count))

// Scale to [-1, 1] range
if peak > 0 {
    var scale = 1.0 / peak
    vDSP_vsmul(samples, 1, &scale, &samples, 1,
               vDSP_Length(samples.count))
}

Without normalization, quiet microphones produce low-amplitude samples that Whisper interprets as silence, triggering hallucinations instead of transcription.

Hotkey Engineering

A voice transcription tool lives or dies by how fast it responds to the trigger key. Two approaches exist on macOS for detecting global keypresses:

CGEventTap

Low-level Core Graphics event tap. Requires Input Monitoring permission. Can silently stop working when the app's own windows have focus.

NSEvent monitors

AppKit-level global + local monitors. Requires only Accessibility permission. Works reliably across all app states.

We use NSEvent monitors — a globalMonitorForEvents for keypresses in other apps and a localMonitorForEvents for keypresses in our own popover or settings window. This is a well-established pattern for macOS menu bar apps.

The subtle part is modifier key detection. Right Shift is a modifier, not a character key. It arrives as .flagsChanged, not .keyDown. We check the raw keycode (60 for Right Shift) on flagsChanged events and maintain toggle state manually — press once to start, press again to stop.

Fighting Hallucinations

Whisper hallucinates. When given silence or near-silence, it generates confident, fluent text that was never spoken. Common outputs include:

  • "Thank you for watching"
  • "Please like and subscribe"
  • "See you in the next video"
  • Repeated phrases ("you you you you you you")
  • Non-ASCII characters for English input

These aren't edge cases. They happen routinely when the user pauses or when background noise is low. The hallucinations come from Whisper's training data — it was trained heavily on YouTube transcripts.

The filter

We run a post-transcription filter that catches three categories:

1
Pattern matching
A list of known hallucination phrases ("thank you for watching", "goodbye", etc.). If the entire transcription matches one of these, discard it.
2
N-gram repetition
Detect 3+ repetitions of the same word or phrase. "You you you you" or "the the the the the the" are discarded. Checks n-grams from 1 to 5 words.
3
Character set validation
For English transcription, reject results where less than half the characters are ASCII. Catches CJK or Cyrillic hallucinations that Whisper sometimes produces.
We initially tried using a technical dictionary as prompt tokens to guide Whisper toward domain-specific vocabulary. This made hallucinations worse — prompt tokens add latency to every decoder step and can bias the model toward generating the prompt itself. We removed them and moved vocabulary correction to post-processing.
